# نشر المشروع مجانًا بالكامل (بدون الاعتماد على جهازك)

هذا الدليل يشرح كيف تجعل الباك-إند والفرونت-إند وقاعدة البيانات والتخزين كلها خارجية ومجانية.
كل خدمة هنا لها خطة مجانية كافية للبداية والتجربة مع عدد محدود من المستخدمين.

## 1. قاعدة البيانات — Neon (Postgres مجاني)

1. أنشئ حسابًا على https://neon.tech
2. أنشئ مشروع جديد (Project)، واختر منطقة قريبة منك.
3. من صفحة الاتصال (Connection Details) انسخ الـ **Connection string** الذي يبدأ بـ `postgres://...`
4. احتفظ به، ستضعه في متغير `DATABASE_URL` لاحقًا.

بديل: Supabase (https://supabase.com) يعطي نفس الشيء مع أدوات إضافية إن أردت التوسع لاحقًا (auth, storage).

## 2. تخزين الوسائط — Cloudflare R2 (10GB مجانًا)

الباك-إند على استضافة مجانية (Render) لا يملك قرصًا دائمًا؛ أي ملفات (صور/صوت/فيديو) تُحفظ محليًا تُمسح عند كل إعادة تشغيل. لذلك نرفعها إلى R2 بدلًا من ذلك.

1. أنشئ حسابًا على https://dash.cloudflare.com
2. من القائمة الجانبية: R2 → Create bucket، اختر اسمًا مثل `movie-studio-media`.
3. **اترك الـ bucket خاصًا (لا تفعّل Public Access)** — الباك-إند يولّد روابط موقّعة (presigned URLs) صالحة لمدة قصيرة عند الطلب فقط، بعد التحقق من أن صاحب الطلب يملك المشروع، لذا لا حاجة لجعل الملفات عامة.
4. من R2 → Manage API Tokens، أنشئ Token له صلاحية القراءة والكتابة على هذا الـ bucket، واحفظ:
   - Account ID
   - Access Key ID
   - Secret Access Key

## 3. الباك-إند — Render (خطة Free)

1. ارفع مجلد المشروع كاملًا إلى مستودع GitHub جديد.
2. من https://render.com أنشئ **New Web Service** واربطه بالمستودع.
3. اختر **Root Directory**: `backend`، واختر **Environment: Docker** (سيستخدم `backend/Dockerfile` تلقائيًا؛ يشمل تثبيت ffmpeg اللازم لتوليد الفيديو والصوت).
4. أضف متغيرات البيئة التالية في تبويب Environment:
   - `JWT_SECRET` = قيمة عشوائية قوية (شغّل `python -c "import secrets; print(secrets.token_hex(32))"` محليًا وانسخ الناتج) — لا تستخدم القيمة الموجودة في `backend/.env` المحلي إن كان قد شُورك مع أي شخص.
   - `DATABASE_URL` = رابط Neon من الخطوة 1
   - `CORS_ORIGINS` = رابط الفرونت-إند على Vercel (تحصل عليه في الخطوة التالية؛ يمكنك تركه مؤقتًا `http://localhost:5173` وتحديثه لاحقًا)
   - `R2_ACCOUNT_ID`, `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_BUCKET` من الخطوة 2
   - `OPENAI_API_KEY` (أو اترك فارغًا لاستخدام Ollama/preview mode)
5. بعد النشر ستحصل على رابط مثل `https://your-app.onrender.com`. جرّبه عبر `/api/health`.

**ملاحظات مهمة عن الخطة المجانية في Render:**
- الخادم "ينام" بعد ~15 دقيقة خمول، وأول طلب بعدها يستغرق حتى 30-50 ثانية لإعادة التشغيل.
- هناك حد زمني ومواردي للطلب الواحد؛ توليد فيديو طويل جدًا (عدة مشاهد + رندر) قد يبطئ أو يفشل تحت ضغط. إن واجهت هذا لاحقًا، الحل هو الترقية لخطة مدفوعة صغيرة (~$7/شهر) وليس تعقيد الكود بطابور Redis/Celery في هذه المرحلة.

## 4. الفرونت-إند — Vercel

1. من https://vercel.com أنشئ مشروعًا جديدًا واربطه بنفس مستودع GitHub.
2. اضبط **Root Directory**: `frontend`.
3. أضف متغير بيئة: `VITE_API_URL` = رابط الباك-إند من Render (مثل `https://your-app.onrender.com`).
4. انشر. ستحصل على رابط مثل `https://your-app.vercel.app`.
5. ارجع إلى إعدادات Render وحدّث `CORS_ORIGINS` ليصبح رابط Vercel هذا بالضبط، ثم أعد نشر الباك-إند.

## بعد التشغيل

المشروع الآن يعمل بالكامل خارج جهازك ومجانًا:
- الفرونت: Vercel
- الباك-إند: Render
- القاعدة: Neon (Postgres)
- الوسائط: Cloudflare R2

عندما يكبر الاستخدام، الترقيات المنطقية بالترتيب: Render خطة مدفوعة صغيرة (يزيل مشكلة النوم والحدود) → إضافة Redis+Celery إن أصبح لديك عشرات الطلبات المتزامنة → Postgres مدفوع عند الحاجة لمساحة أكبر.

## قائمة التحقق قبل الإطلاق (Production Checklist)

- [ ] `JWT_SECRET` في بيئة الإنتاج قيمة عشوائية قوية خاصة بالإنتاج فقط (وليست منسوخة من أي ملف `.env` محلي شُورك أو رُفع بالخطأ).
- [ ] `CORS_ORIGINS` يطابق رابط الفرونت-إند الفعلي بالضبط (وليس `*` أو `localhost`).
- [ ] `DATABASE_URL` يشير إلى Postgres (Neon/Supabase)، وليس SQLite المحلي.
- [ ] بيانات R2 مضبوطة والـ bucket **غير عام** (تحقق أن Public Access معطّل).
- [ ] لا يوجد أي ملف `.env` حقيقي داخل مستودع Git (تحقق من `.gitignore`).
- [ ] عند التشغيل بأكثر من عامل (worker) واحد أو أكثر من نسخة (instance)، استبدل مخزّن Rate Limiting الافتراضي (في الذاكرة) بمخزن مشترك مثل Redis عبر `slowapi`، وإلا فالحد الفعلي يتضاعف مع كل عامل/نسخة إضافية.
- [ ] راجع حدود Rate Limiting الحالية (تسجيل الدخول 10/دقيقة، التسجيل 5/دقيقة) وعدّلها إذا لزم حسب حجم الاستخدام المتوقع.

## قيود معروفة (Known Limitations)

- **مخزّن Rate Limiting**: في الذاكرة حاليًا، مناسب لعامل واحد (worker) فقط — راجع البند أعلاه عند التوسع.
- **روابط الوسائط**: تُولَّد كرموز صالحة لعشرين دقيقة تقريبًا؛ إن ترك المستخدم الصفحة مفتوحة لفترة أطول قد يحتاج لإعادة تحميلها (يحدث تلقائيًا مع أي تحديث للصفحة).
- **قاعدة البيانات المحلية (SQLite)**: مناسبة للتجربة فقط؛ استخدم Postgres في الإنتاج كما هو موضح أعلاه.
